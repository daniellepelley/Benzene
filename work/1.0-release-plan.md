## 1. The release decision (settled)

**Big-bang release, not a narrow-core release.** The core packages ship 1.0.0 **together with**
the AWS and Azure packages, because nobody adopts the core without a cloud host — the core alone
has no audience. This raises the bar: it means the AWS and Azure surfaces (ingress **and** egress)
and the capability packages must all be either genuinely 1.0-ready or **honestly scoped**, because
none of them get to hide behind a `-preview` tag.

Consequence: the correct 1.0 is **not** "ship less." It is "ship the whole primary surface, with
documentation that tells the exact truth about what each package does and does not do, and with
the deliberate non-goals stated as non-goals rather than discovered as gaps."

Scope of this release: **AWS, Azure, ASP.NET Core, self-hosted.** Google Cloud and Cloudflare are
explicitly **out of scope** for 1.0 — they are lesser offerings today and should be marked
experimental/community and held out of the 1.0 cut (or clearly labelled), not blocking it.

## 2. Design philosophy — the decision framework

Every "should Benzene do X?" question below is answered by these principles. They are the reason
some findings are *build-it* and others are *document-it-as-a-non-goal*.

1. **Abstract at the business-logic boundary, never at the transport.** Benzene's abstraction
   lives in the message handlers — write your handler once, host it anywhere. It deliberately does
   **not** abstract the transport itself. This is the opposite of Dapr: Dapr hides the third party
   behind a generic interface (a "queue", a "state store"), and in doing so hides the third party's
   *core capabilities*. If you abstract SQS behind a generic queue, you lose the SQS-specific
   features that were the reason to choose SQS. Benzene never does this — you keep full access to
   the underlying SDK and all its cloud-native capabilities. We abstract *where the business logic
   is*, so the handler is portable, while the transport stays fully itself.

2. **Bring your own code for anything that isn't a transport.** Benzene does **not** get involved
   in database access. A database is not a transport. This is why Cosmos DB has a change-feed
   *ingress* (the feed is an event stream) but **no egress** (writing to Cosmos is database access —
   your handler uses `CosmosClient` directly). Same for Blob storage as a store, EF Core, etc.
   Benzene provides the pipe in; persistence is yours.

3. **If we don't ship a cloud offering out of the box, rolling your own is easy and first-class.**
   The extension model (custom middleware, custom pipes into Benzene, the getter/setter mapper
   pattern) is a supported public path, not an escape hatch. "Benzene doesn't have an adapter for
   X" should always be answered by "…and here's the 20-line middleware that adds it," documented.

4. **Let people use the libraries they already know.** Where a best-in-class library exists (Polly
   for resilience, FluentValidation for validation, the cloud SDKs), Benzene's job is to give it a
   clean place to plug into the pipeline — **not** to wrap it in a Benzene abstraction that hides
   its capabilities (see principle 1). A "Benzene.X integration" package exposes X, it doesn't
   replace X.

5. **Honesty over false capability.** For a hexagonal framework, the package documentation *is*
   the API contract — it is the first thing an evaluator reads. Docs that claim capabilities the
   code doesn't have are the single biggest launch risk. Where Benzene deliberately doesn't solve
   something (cross-process idempotency, database access, transport abstraction), we say so plainly
   **and** show how to solve it outside Benzene. A stated non-goal builds trust; a discovered gap
   destroys it.

6. **Some problems cannot be solved at runtime inside Benzene, and that's fine.** Benzene instances
   are independent processes (e.g. separate Lambda invocations) that don't know about each other.
   Problems requiring cross-instance coordination (distributed idempotency, exactly-once) can't be
   solved by Benzene alone; bolting on shared state can just move the problem to a race condition.
   The honest answer is an external store with atomic semantics plus naturally-idempotent handler
   design — documented, not pretended away.

## 3. The assessment in brief (the six convergent themes)

All five lenses independently converged on these. Full evidence is in §7 (Appendix: findings by
lens).

- **T1 — The docs oversell the code.** Verified false claims across many `CLAUDE.md`/package docs:
  `Benzene.Resilience` advertises "Polly, retry, circuit breaker, timeout, bulkhead" but ships
  retry-only with no Polly; `Benzene.Aws.Sqs` claims FIFO/dedup/batch that don't exist; ApiGateway
  docs claim HTTP API v2 and reference a deleted `Cors` type; `Client.Http`/`Cache.Redis`/`Results`
  docs name capabilities/types not in code. **The #1 launch risk. Highest leverage, lowest effort.**
- **T2 — The release machinery can't cut a 1.0.** License metadata says Apache-2.0 while the repo
  is MIT (verified: `Directory.Build.props:8` vs `LICENSE`/README badges); committed repo-root
  junk leaking an internal name (`Output/…PlatformPedalCore…`, `86 lines changed`, neither
  gitignored); the publish pipeline hardcodes `-alpha` and a four-segment version so it can never
  emit `1.0.0` (`deploy-benzene.yml:45`); examples aren't compile-gated except Azure; GitHub-broken
  extensionless doc links; missing SECURITY/COC/templates; thin NuGet metadata, no SourceLink.
- **T3 — API-freeze warts become permanent.** Oversized public surface in `*.MessageHandlers`
  (impl types public — can't be made `internal` post-1.0 without a major bump); naming warts
  (`MessageMessageHandlerResultSetter` ×20, `IRequestMapperThunk`); `IDependencyWrapper<T>` already
  `[Obsolete]` in the root package. Big-bang makes these **release-blocking**, since everything
  freezes at once.
- **T4 — Capability packages are abstraction+in-memory-only.** Idempotency (in-memory single-
  process only), Saga (no durable resume), SchemaRegistry (compat checker is a textual no-op),
  Resilience (retry-only). Under big-bang these can't be `-preview` — they must be **honestly
  scoped** per principles 5 & 6.
- **T5 — Egress is the real functional gap.** Azure has **no outbound story at all**; AWS clients
  exist but are lumped in one package (pulling in SQS to use SNS) and Step Functions is a stub.
  This is genuine missing capability → **build it** (§5).
- **T6 — Async-transport observability is weakest where Benzene differentiates.** Distributed
  tracing is HTTP-only; SQS/SNS/Kafka/Event Hub inbound trace extraction isn't implemented;
  `invocationId` dropped in per-message sub-pipelines. The queue transports that *are* the pitch
  have the poorest correlation.

**Genuinely strong (showcase, don't re-litigate):** core middleware pipeline; `Benzene.Core.Versioning`;
results/media-negotiation; `Benzene.Auth.OAuth2` authN (RFC 8725 algorithm-confusion protection,
no token oracle); SQS partial-batch-failure; DynamoDB/Kinesis streaming semantics; the Azure
trigger matrix + Event Grid CloudEvents + Service Bus explicit-ack + self-hosted workers + managed
identity; k8s health probes; the self-hosted worker concurrency/lifecycle rewrite; the DI-scope
leak fix; System.Text.Json default.

## 5. Egress design (Tier 2 detail)

The organising rule (principle 2): **every messaging *transport* gets an egress package that mirrors
its ingress; things that aren't transports (databases, stores, schedulers) get no egress — you bring
your own code.**

### 5.1 AWS client segregation

Today all AWS outbound clients live in one package, `src/Benzene.Clients.Aws` (SQS, SNS,
EventBridge, Lambda-invoke, StepFunctions). Using the SNS client therefore drags in the SQS client
and its SDK. Split along the existing folder boundaries into per-transport packages, mirroring how
ingress is split (`Benzene.Aws.Lambda.Sqs`, `.Sns`, …):

| New package (proposed) | Contains (from today's `Benzene.Clients.Aws/…`) |
|---|---|
| `Benzene.Clients.Aws.Sqs` | `Sqs/*` — SQS send client + outbound converter + health check |
| `Benzene.Clients.Aws.Sns` | `Sns/*` — SNS publish client + converters |
| `Benzene.Clients.Aws.EventBridge` | `EventBridge/*` — PutEvents client |
| `Benzene.Clients.Aws.Lambda` | `Lambda/*` — Lambda invoke client |
| `Benzene.Clients.Aws.StepFunctions` | `StepFunctions/*` — StartExecution client (scope per 2.5) |
| `Benzene.Clients.Aws.Core` *(if needed)* | any shared outbound wiring currently in `Clients.Aws/Extensions.cs` |

Shared transport-agnostic outbound infrastructure (routing, correlation, trace-context propagation)
stays in `Benzene.Clients` (the existing base — the outbound analogue of `Benzene.Aws.Lambda.Core`).
`Benzene.Clients.Aws` becomes either a thin meta-package that references all five (for people who
genuinely want everything) or is retired — **naming/meta-package call for the maintainer.** Each new
package pins only the one AWS SDK it needs. *Done when: referencing `Benzene.Clients.Aws.Sns` pulls
in `AWSSDK.SimpleNotificationService` only, not `AWSSDK.SQS`.*

### 5.2 Azure egress packages (currently absent entirely)

Build the outbound half so an Azure Benzene service can publish with the same correlation/trace
propagation the AWS path gets for free. 1:1 with ingress **where the target is a transport**:

| Azure transport | Ingress (exists) | Egress (build) | Notes |
|---|---|---|---|
| Service Bus | `Benzene.Azure.Function.ServiceBus` + `Benzene.Azure.ServiceBus` | **`Benzene.Clients.Azure.ServiceBus`** | `ServiceBusSender` to queue/topic; set the `topic` app-property Benzene routes on |
| Event Hubs | `Benzene.Azure.Function.EventHub` + `Benzene.Azure.EventHub` | **`Benzene.Clients.Azure.EventHub`** | `EventHubProducerClient` |
| Event Grid | `Benzene.Azure.Function.EventGrid` | **`Benzene.Clients.Azure.EventGrid`** | `EventGridPublisherClient`; support both EG-schema and CloudEvents |
| Queue Storage | `Benzene.Azure.Function.QueueStorage` | **`Benzene.Clients.Azure.QueueStorage`** | `QueueClient.SendMessage` (Benzene envelope body) |
| Kafka (over Event Hubs) | `Benzene.Azure.Function.Kafka` | **`Benzene.Clients.Azure.Kafka`** *(or fold into EventHub)* | producer over the Kafka endpoint — decide whether it's its own package or covered by the Event Hub producer |
| HTTP | `Benzene.Azure.Function.AspNet` | *(cross-cloud)* `Benzene.Client.Http` | already exists; not Azure-specific |
| **Blob Storage** | `Benzene.Azure.Function.BlobStorage` | **none** | writing a blob is *storage access*, not a transport — bring your own `BlobClient` (principle 2) |
| **Cosmos Change Feed** | `Benzene.Azure.Function.CosmosDb` + worker | **none** | Cosmos is a **database, not a transport**. No egress. Persisting to Cosmos is your handler's own `CosmosClient` code (principle 2) |
| **Timer** | `Benzene.Azure.Function.Timer` | **none** | a scheduler trigger — nothing to publish |

Shared transport-agnostic outbound infra stays in `Benzene.Clients`; a `Benzene.Clients.Azure.Core`
holds any Azure-shared wiring / the `TokenCredential` managed-identity seam (consistent with the
ingress factories). Each package pins only its one Azure SDK. *Done when: an Azure handler can
publish to Service Bus/Event Hubs/Event Grid/Queue Storage through Benzene with correlation + trace
propagation, and the "no egress for Blob/Cosmos/Timer — bring your own code" rule is documented on
each of those ingress packages.*

### 5.3 The symmetry principle, documented

Publish an **ingress↔egress matrix** (both clouds) as a first-class doc page, with the deliberate
asymmetries (Blob/Cosmos/Timer/S3-notifications have no egress because they aren't transports)
stated as *design*, not omission. This doc is itself a strong expression of the philosophy and a
good adoption asset.

## 6. Capability-honesty matrix (Tier 1.3 — the deliverable shape)

A single table (README-linked, and linked from each capability package) with three columns:
**What Benzene does · What it deliberately doesn't (and why) · How to solve it outside Benzene.**
Seed rows:

| Capability | Benzene provides | Deliberately NOT (why) | Solve it via |
|---|---|---|---|
| Idempotency | Seam + in-memory single-process store; atomic-claim middleware | Cross-instance dedup — independent processes can't coordinate at runtime without external state, and shared state can just relocate the race (principle 6) | External store with atomic conditional write (DynamoDB conditional put / Redis SETNX) + naturally-idempotent handlers; reference cookbook |
| Resilience | Retry w/ exponential backoff middleware | Circuit breaker / timeout / bulkhead — best served by Polly, which we won't hide (principle 4) | Your own Polly `ResiliencePipeline` in middleware (± a thin `Benzene.Resilience.Polly` pipe — Tier 3.2) |
| Saga / workflow | In-process compensation-based saga with LIFO rollback | Durable crash-resume (closures can't be rehydrated) | Step Functions / a durable workflow engine for crash-durable orchestration |
| Schema evolution | Confluent wire-format codec; registry seam | Structural Avro/JSON backward-compat checking in-box (checker is byte-identical only) | A real schema-registry server, or supply your own `ISchemaCompatibilityChecker` |
| Database access | *(nothing — by design)* | Any DB/store abstraction (a DB isn't a transport — principle 2) | Your handler's own SDK/ORM (`CosmosClient`, EF Core, Dapper, …) |
| Transport features | Full access to the underlying SDK/message | Hiding transport-specific capabilities behind a generic interface (the Dapr anti-pattern — principle 1) | Use the raw SDK feature directly; the context exposes the native message |
| Multi-tenancy | *(nothing today)* | — (candidate post-1.0) | Roll a tenant-resolver middleware + scoped holder (documented pattern) |
| Distributed tracing | W3C trace context on HTTP transports; OTel exporter-agnostic | Async-transport trace extraction today (Tier 3.5 may close this) | Until 3.5 lands: propagate correlation via message headers |

## 8. How we work this plan

- Iterate tier by tier; within a tier, order by the done-when that unblocks the most.
- Every closed checkbox is verified against code (build/test/read), never against a doc.
- Merge each unit of work to `main` behind a green build + the (new) examples gate.
- Keep this document current — it is the single source of truth; the older `work/*roadmap*` and
  `*checklist*` docs are history and must not be cited as status.
- Strategic calls that are the maintainer's (package-tier meta-naming, the naming-wart freeze,
  whether async-tracing and the Polly package land in 1.0 vs. fast-follow) are flagged inline — we
  decide them as we reach them, not silently.
