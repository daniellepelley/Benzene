# Benzene 1.0 Readiness Checklist (2026-07-17)

**Status:** Consolidated cross-reference, not a new audit. This document exists because the
individual `work/*-roadmap-1.0.md` and `work/1.0.0-release-status.md` documents each track their
own slice (a package family, or a specific process step) and none of them, on their own, answers
"is Benzene actually ready to tag `v1.0.0` and publish a stable NuGet release?" This is that answer,
as of today, sourced from those documents plus direct repo checks (`version.txt`, `git tag`).

**Bottom line up front: not yet.** `version.txt` is still `0.0.2`; there is no git tag in the repo
(`git tag --sort=-creatordate` returns nothing); every NuGet release published so far is an
alpha-suffixed prerelease (`deploy-benzene.yml` computes `{version.txt}.{build}-alpha` at publish
time). The individual process steps below are further along than that headline suggests, but
several concrete items block an actual 1.0 tag.

## What's actually done

- **API surface review** (`work/api-surface-review.md`, `work/1.0.0-release-status.md` Step 1) —
  the "Must Fix (Blocking)" list from the original 2026-07-08 review was re-audited on 2026-07-14
  and everything on it is resolved: no `ToDelete` folder, no test-helper code shipping inside
  production packages (a repo-wide sweep on 2026-07-14 caught 3 more leaks the first pass missed,
  since fixed), `BenzeneWorkerStartup2` renamed to `BenzeneWorkerBuilder`, the dead/unreachable
  `AddScoped<T>(T instance)` overload removed.
- **XML documentation** — 100% of public APIs across the 5 core packages documented, XML doc
  generation enabled in build config.
- **Versioning policy** — `VERSIONING.md` and `CHANGELOG.md` exist with a real semver policy and a
  genuine `## [Unreleased]` history (not backfilled placeholder text).
- **Test coverage on core packages** — `Benzene.Abstractions` 100%, `Benzene.Core.Middleware` 95.5%,
  `Benzene.Core` 98.6%, `Benzene.Http` 95.6% line coverage (per `work/1.0.0-release-status.md` Step
  3; not independently re-verified for this checklist — no local .NET SDK available in this
  environment, per `AGENTS.md`).
- **CI now runs every test project that exists in the solution** — as of today,
  `Benzene.Grpc.Test`, `Benzene.Mesh.Test`, and `Benzene.Conformance.Test` were wired into
  `build-benzene.yml` (previously compiled by CI but never actually executed). Combined with the
  pre-existing `Benzene.Core.Test`/`Benzene.Aws.Tests`/`Benzene.Integration.Test` jobs, there is no
  longer a test project in the repo that CI silently skips running.
- **AWS and Azure package families** — both roadmaps (`aws-roadmap-1.0.md` v1.9,
  `azure-roadmap-1.0.md` v1.8) report their tracked P0 items resolved as of the 2026-07-14 audit
  pass (CORS parity, dependency-crisis fixes, XML docs, real test coverage numbers replacing
  earlier false "zero tests" claims).
- **Breaking-change communication** — `docs/migration-alpha-to-1.0.md` exists and documents the
  renames found during the docs/examples audit.

## What blocks an actual 1.0 tag

1. **No git tag, no numbered release, still on `0.0.2`.** `work/1.0.0-release-status.md`'s own
   "Step 6: Publish" checklist (bump versions, build/test, publish to NuGet, cut a GitHub Release,
   drop the "prerelease" README badges) is explicitly marked **Not Started**. This is the actual
   blocking gap — everything else is prerequisite work, not the release itself.
2. **This project's own status-tracking has a documented false-positive history.**
   `1.0.0-release-status.md` originally claimed "Test Helpers Extraction: 100% Complete," which was
   wrong — a later direct code audit (2026-07-14) found 3 real leaks the first pass missed. The same
   document explicitly says its own "Step 6: Publish" readiness assessment "should be treated with
   the same skepticism until independently re-checked." None of the `work/*-roadmap-1.0.md`
   documents carry a status better than **"DRAFT for Review"** (aws, azure, observability, dx) or
   **"first entry"/"IN PROGRESS"** (performance, service-mesh) — none has been formally closed out
   by anyone other than the same automated process that wrote it. Before tagging 1.0, someone other
   than the tool that did the work should independently re-verify at least the AWS/Azure/core
   claims, given the track record of self-corrected false completeness claims in this exact
   process.
3. **Observability package family lost 6 of 12 tracked packages outright**
   (`work/observability-roadmap-1.0.md`) — `Benzene.Microsoft.Logging`, `Benzene.Serilog`,
   `Benzene.Log4Net` and others were deleted when the custom logger stack was replaced with
   `Microsoft.Extensions.Logging`. That's a legitimate scope cut, not a gap by itself, but it means
   the roadmap's own package inventory is stale and needs a fresh pass before anyone can sign off on
   "observability is 1.0-ready."
4. **GCP integration is not part of any 1.0 scope decision.** `Benzene.GoogleCloud.*` is
   Phase 0 of 5 (`work/google-cloud-roadmap-1.0.md`) — HTTP-trigger-only, no Pub/Sub. This is fine
   *if* GCP is explicitly scoped out of 1.0 (matching the "Conservative Approach" release strategy
   below, which doesn't mention GCP at all) — but that exclusion isn't stated anywhere as a decision,
   it's just an absence. Worth an explicit line in the release scope so it doesn't read as an
   oversight.
5. **Kinesis and ServiceBus partial-failure containment gaps remain** (see
   `work/batch-failure-handling.md`'s "Deliberately deferred" table) — `Aws.Lambda.Kinesis` has no
   per-record result tracking at all (confirmed to need a streaming-abstraction redesign, not a
   small fix), and `Azure.Function.ServiceBus` lacks true per-message `ServiceBusMessageActions`
   ack/abandon/dead-letter. Neither blocks 1.0 *if* the "Conservative Approach" release strategy
   below is followed (both packages are "Keep at 0.9.x-preview," not "Ship at 1.0.0"), but this
   needs to be an explicit decision, not an implicit one.
6. **`Benzene.Clients*` outbound story is self-flagged as underdeveloped**
   (`work/benzene-clients-vision.md`) — needs a redesign before any part of that surface can carry a
   1.0 semver guarantee.
7. **Package scope decision itself was never finalized.** `1.0.0-release-status.md`'s "Release
   Strategy" section proposes a **Conservative Approach**: ship only 5 foundational packages at
   1.0.0 (`Benzene.Abstractions`, `Benzene.Abstractions.Middleware`, `Benzene.Core`,
   `Benzene.Core.Middleware`, `Benzene.Http`), keep everything else — including all AWS/Azure
   adapters — at `0.9.x-preview`. That's listed as "RECOMMENDED," not confirmed. Given how deep the
   AWS/Azure package coverage actually is (see the platform assessment from earlier today), shipping
   *only* the 5 core packages at 1.0 while leaving every real transport adapter at preview would be
   a strange signal to send adopters — this is a maintainer call, not something to resolve
   unilaterally here.
8. **Single-maintainer bus factor** (from today's broader platform assessment, not from any
   `work/` doc) — 356 commits, all from one human author plus heavy AI-assisted commits, no
   external contributors visible in git history, no tagged releases yet. Not a code blocker, but a
   real factor in whether outside adopters should treat a 1.0 tag as a durable commitment. A
   `CONTRIBUTING.md` was added today as one small step toward lowering the barrier to outside
   contribution; it doesn't by itself change the bus-factor picture.

## What I am and am not doing about this

Per the standing instruction in this session: **no git tag will be created and no non-alpha NuGet
release will be published without explicit user confirmation** — that's a business/versioning
decision, not a code-implementation task, and it's also a hard-to-reverse action (a published NuGet
version can't be unpublished, only deprecated) in the "check first" category this project's own
workflow conventions call for. The items above that *are* concrete code/doc work (Kinesis,
ServiceBus, Clients redesign, GCP next phase) are tracked as their own separate work items and will
be implemented independently of any 1.0-tagging decision.
