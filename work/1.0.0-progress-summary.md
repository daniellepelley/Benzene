## Recommended 1.0 Strategy

### Conservative Approach (RECOMMENDED)

**Ship at 1.0.0:**
- Benzene.Abstractions
- Benzene.Abstractions.Middleware
- Benzene.Core
- Benzene.Core.Middleware
- Benzene.Http

**Keep at 0.9.x-preview:**
- Benzene.Abstractions.MessageHandlers
- Benzene.Core.MessageHandlers
- Benzene.Aws.Lambda.Core
- Benzene.Aws.Lambda.ApiGateway
- Benzene.Aws.Lambda.Kafka (0.5.x - newer)

**Rationale:**
1. Core middleware infrastructure is solid and stable
2. Message handling layer has ToDelete issues and test helper pollution
3. AWS adapters depend on AWS SDK versions which may evolve
4. Gives users stable foundation while maturing other layers

## Cross-Cutting Concerns Found

### 1. Zero XML Documentation
- **Impact:** CRITICAL - No IntelliSense help for users
- **Scope:** ALL packages (200+ public types)
- **Effort:** 20-30 hours for core 5 packages

### 2. Over-Exposed APIs
- Many implementation details are public (should be internal)
- Examples: `CacheMessageHandlersFinder`, `UrlMatcher`, utility classes
- **Impact:** Harder to evolve, larger perceived API surface

### 3. Naming Inconsistencies
- Double "Message": `MessageMessageHandlerResultSetter`
- File names ≠ class names
- Inconsistent suffixes: "Builder" vs "Build", "Getter" vs "Get"

### 4. Test Code in Production
- 14 TestHelpers files ship with production packages
- Increases DLL size, pollutes API

### 5. "ToDelete" Folder in Production
- Public interfaces in folder named "ToDelete"
- Unclear intent, unprofessional for 1.0

## Key Recommendation

**Do NOT ship everything at 1.0 simultaneously.** The message handling and AWS adapter layers have too many unresolved issues. Ship the solid core (5 packages) at 1.0, keep the rest at preview, and promote them to 1.0 after cleanup over the next 3-6 months.

This gives users a stable foundation to build on while maintaining flexibility to improve the less-mature layers.
