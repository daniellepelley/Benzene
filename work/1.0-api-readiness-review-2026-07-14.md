### 1b. Versioned public type name: `BenzeneWorkerStartup2`

`src/Benzene.SelfHost/BenzeneWorkerBuilder.cs` exposes `public class BenzeneWorkerStartup2`. A `2`
suffix in a public type name is a code smell to lock in at 1.0 — it implies a superseded v1. Rename
to `BenzeneWorkerStartup` (or make it `internal` if it's an implementation detail behind
`IBenzeneWorkerStartup`). Check for a leftover `BenzeneWorkerStartup` v1 to remove.

## 2. Decisions the maintainer needs to make (not mechanical)

### 2a. `[Obsolete]` APIs — ship or remove at 1.0

Three obsolete members, each with a clear "superseded by" message:
- `src/Benzene.Diagnostics/Correlation/Extensions.cs` — `UseCorrelationId` (→ W3C `traceparent`)
- `src/Benzene.Aws.Lambda.Core/LogContextBuilderExtensions.cs` — two methods (→ `UseBenzeneEnrichment`)

**Recommendation:** keep them `[Obsolete]` (not error) for the 1.0 line — the messages are good and
removal is a breaking change you can defer to 2.0. But 1.0 is the last free moment to remove them if
you'd rather not carry them. Pick one and note it in the migration guide.

### 2b. Per-package 1.0 vs `-preview` tiering (checklist §1)

The newest surfaces are materially less battle-tested than `Benzene.Http`/`Benzene.Aws.Lambda.*`:
gRPC (`Benzene.Grpc*`), the streaming fan-in (`Benzene.Core.Middleware/Streaming`, unmerged on PR #2),
schema-compatibility (`Benzene.Schema.OpenApi/Compatibility`, unmerged), and the Azure isolated-worker
/ ServiceBus packages. Consider shipping the mature core at `1.0.0` and keeping the newest at a
`-preview` tag rather than committing to their signatures for a year.

### 2c. Version path

`version.txt` is `0.0.2`; `CHANGELOG.md` has a maintained `[Unreleased]` section (good). The bump to
`1.0.0` is a deliberate release step — out of scope for autonomous change.
