# 1.0 API-Freeze Proposal — decisions for the maintainer

**Created:** 2026-07-18 · **Status:** AWAITING MAINTAINER DECISIONS · Plan ref: Tier 1.2 in
`work/1.0-release-plan.md`.

## Why this is now-or-never

A 1.0 release freezes the public API under semver. Because the release is **big-bang** (core +
AWS + Azure ship 1.0 together), *everything public freezes at once*. After 1.0:

- A public type **cannot be made `internal`** without a major version bump.
- A public member **cannot be renamed** without a major version bump.

So anything in the public surface that is really an implementation detail, or is misnamed, has to
be fixed **before** the tag or carried for the entire 1.x line. This document lists the specific
calls, each with a recommendation and the evidence. **These are the maintainer's decisions** — I've
implemented nothing here; I'll execute whichever you approve.

Good news up front: **Decision D is already resolved** — `IDependencyWrapper<T>` and *every*
`[Obsolete]` member are already gone from `src/` (verified: zero hits). One less thing.

## Decision A — Reduce the public surface of the MessageHandlers packages *(highest value)*

**Evidence.** `Benzene.Abstractions.MessageHandlers` exposes **41** public types and
`Benzene.Core.MessageHandlers` **58** (99 total). Many are internal plumbing that a consumer never
references — freezing them commits us to their shape forever. Clear internal candidates
(implementation details, not API a user or extension author calls):

- The handler-discovery finders: `ReflectionMessageHandlersFinder`, `CacheMessageHandlersFinder`,
  `CompositeMessageHandlersFinder`, `DependencyMessageHandlersFinder`.
- The dispatch plumbing: the **non-generic** `IMessageHandler` (the transport-facing dispatch
  entry point — *not* the `IMessageHandler<TRequest,TResponse>` users implement),
  `IRequestMapperThunk` + `RequestMapperThunk<TContext>`, `IMessageHandlerFactory`,
  `MessageHandlerNoResultWrapper`, `PipelineMessageHandlerWrapper`.
- Response-mapping internals: `BenzeneMessageResponseAdapter`, `DefaultResponsePayloadMapper`,
  `DefaultResponseStatusHandler`.

**What stays public** (the real API): `MessageAttribute`, `MessageRouter`/`MessageRouterBuilder`,
`MessageHandlerMiddleware`, `IMessageHandler<TRequest,TResponse>` / `IMessageHandler<TRequest>`
(the interfaces users implement), the `Use*`/`Add*` extension methods, `MessageHandlerContext`,
`PresetTopicMiddleware`, `TransportMiddlewarePipeline`, `IMessageHandlersFinder` (the seam), etc.

**Recommendation:** do a type-by-type pass and mark the plumbing `internal` (using
`InternalsVisibleTo` for the test project where needed). This is the single most valuable freeze
action — it shrinks the forever-surface and, as a bonus, resolves Decision C for free.
**Risk:** low — these types aren't part of any documented usage; the compile check + full test
suite will catch anything that actually depended on them being public.

**Your call:** ☐ approve the internal-candidate list above (I'll bring the exact per-type list for
sign-off before editing) · ☐ keep some public (name them) · ☐ defer (accept the 99-type surface).

## Decision B — Rename the `MessageMessageHandlerResultSetter` family *(cosmetic but permanent)*

**Evidence.** The double-`Message` typo is baked into **19 distinct public types** — one per
transport (`SqsMessageMessageHandlerResultSetter`, `KafkaMessageMessageHandlerResultSetter`,
`ServiceBusMessageMessageHandlerResultSetter`, …) plus the bases
(`MessageMessageHandlerResultSetterBase`, `ResponseMessageMessageHandlerResultSetterBase`,
`DefaultMessageMessageHandlerResultSetterBase`). It's visible in every transport package.

**Options:**
1. **Rename** to a single `Message` (e.g. `SqsMessageHandlerResultSetter`) — mechanical, touches
   ~19 files across every transport package, breaks anyone referencing them by name (acceptable
   pre-1.0; impossible post-1.0). Cleanest long-term.
2. **Make them `internal`** — most of these are registered by the framework and never referenced by
   name in user code, so they may not need to be public at all (overlaps Decision A). If they can
   be internal, the name stops mattering.
3. **Accept it** — keep the wart forever.

**Recommendation:** first check (as part of Decision A) whether each `...ResultSetter` needs to be
public at all. Those that don't → `internal` (name becomes moot). Those that must stay public →
rename now. Likely outcome: most go internal, few need renaming.

**Your call:** ☐ internal-where-possible + rename-the-rest (recommended) · ☐ rename all · ☐ accept.

## Decision C — The dispatch interface naming (`HandlerAsync`, "Thunk")

**Evidence.** The non-generic dispatch interface has two warts:
`IMessageHandler.HandlerAsync(IRequestMapperThunk)` — the method is `HandlerAsync` (a typo; every
other method in the codebase is `HandleAsync`), and its parameter type carries the `Thunk` jargon.
**Crucially, this is the framework dispatch surface, not the interface users implement** (users
implement `IMessageHandler<TRequest,TResponse>.HandleAsync`).

**Recommendation:** this is **resolved for free by Decision A** — if the non-generic
`IMessageHandler`, `IRequestMapperThunk`, and `IMessageHandlerFactory` go `internal`, the typo and
the jargon leave the public surface entirely and need no rename. If you decide to keep any of them
public, then rename `HandlerAsync` → `HandleAsync` and consider dropping "Thunk" (e.g.
`IDeferredRequestMapper`) before the freeze.

**Your call:** ☐ resolve via Decision A (internalize) · ☐ keep public but rename · ☐ accept.

- **Package tiering.** Under big-bang, every package ships 1.0 together, so there's no
  stable-vs-`-preview` split to decide — but that raises the honesty bar (the capability packages
  must be truthfully documented, per Tier 1.1 + the capability matrix). The `*.TestHelpers`,
  `Benzene.Testing`, and `Benzene.Tools` packages are user-facing test support and legitimately
  public; the only question is whether to publish all 121 packages on day one or hold the
  peripheral/experimental ones (Google/Cloudflare) — a **discoverability** decision, not an
  API-freeze one. Recommend holding Google/Cloudflare out of the 1.0 cut (out of scope this round).
