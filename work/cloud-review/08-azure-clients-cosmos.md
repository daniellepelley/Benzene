## Azure clients + Cosmos DB

Scope: `src/Benzene.Clients.Azure.EventGrid`, `src/Benzene.Clients.Azure.QueueStorage`, `src/Benzene.Azure.CosmosDb`, cross-checked against the ingress `Benzene.Azure.Function.QueueStorage` and learn.microsoft.com.

Verified SDK facts:
- **Queue Storage:** `Azure.Storage.Queues` v12 does **not** encode by default (`QueueMessageEncoding.None`); v11 and **Azure Functions default to Base64**. MS recommends setting `QueueClientOptions.MessageEncoding = Base64` for Functions interop.
- **Event Grid:** `EventGridEvent` requires id, eventType, subject, eventTime, dataVersion, data (id/eventTime auto-generated by ctor). CloudEvents requires specversion, type, source, id (auto).

---

### Event Grid

**[DIVERGENCE] CloudEvent `subject` unset; classic `subject` overloaded with `eventType`** (Severity: Low)
- CloudEvents path sets `type`=topic, `source`=caller string, never sets `subject`. Classic path sets **both** `subject` and `eventType` to the same topic string. `subject` conventionally identifies the resource/path and is a first-class subscription filter dimension (`subjectBeginsWith`/`subjectEndsWith`) — overloading it collapses that filter axis. Required fields are all populated, so not a validity bug. Documented deliberate choice; note the lost subject-filter capability.

**[MISSING] No batch publish** (Severity: Medium)
- `EventGridClientMiddleware` calls `SendEventAsync` (singular) once per send. `SendEventsAsync(IEnumerable)` batches many events / ~1 MB per HTTP call. Offer a batch overload.

**[MISSING] No Event Grid domain or namespace-topic support** (Severity: Low/Medium)
- Targets a single classic custom topic/domain via a fixed `source`; never sets `EventGridEvent.Topic`. Event **domains** require per-event topic targeting; Event Grid **Namespaces** (MQTT/pull) use a different client (`EventGridSenderClient` in `Azure.Messaging.EventGrid.Namespaces`). Allow per-message topic override for domains; document namespaces out of scope.

**Verdict — Event Grid:** correct and schema-faithful for single-event classic-topic/CloudEvents publishing; gaps are batch, domains, namespace topics — not correctness.

---

### Queue Storage

**[DIVERGENCE / WRONG-APPROACH] Message encoding unspecified — egress defaults to plain text while the Benzene Functions ingress defaults to Base64** (Severity: High) — *the load-bearing finding*
- `QueueStorageClientMiddleware` calls `SendMessageAsync(context.MessageText)` on a caller-built `QueueClient`. With no `QueueClientOptions.MessageEncoding`, v12 default is `None` → JSON envelope goes on the queue as plain UTF-8. Neither code nor CLAUDE.md mentions encoding.
- This package's stated purpose is to feed `BenzeneMessageQueueStorageHandler` on the ingress side — which is the Functions `[QueueTrigger] string` binding, defaulting to **Base64**. So Benzene's own two halves default to **opposite encodings**: plain-text egress → Base64-expecting trigger, which fails to decode the JSON and garbles/dead-letters the message. Silent, config-dependent, undocumented — the classic footgun.
- Recommendation: Document the encoding contract prominently and either instruct callers to build the `QueueClient` with `MessageEncoding = Base64` for a Functions consumer, or match the trigger default. At minimum a CLAUDE.md warning.

**[MISSING] No 64 KB message-size guard; envelope inflates payload** (Severity: Medium)
- Serializes payload to a string, then embeds it (escaped) inside a `BenzeneMessageRequest` envelope serialized again — real expansion — with no size check. Queue messages cap at 64 KB (~48 KB original bytes after Base64). Oversize throws (surfaced as generic `ServiceUnavailable`). Note the limit; consider a pre-send size check.

**[MISSING] No visibility-timeout or TTL on send** (Severity: Low)
- Only `SendMessageAsync(string)` is used; the overload with `visibilityTimeout`/`timeToLive` supports delayed-visibility (scheduled) messages and per-message TTL. Optional overload.

**Verdict — Queue Storage:** functionally sends, but the unspecified/plain default encoding is a real same-stack interop footgun against Benzene's own Functions ingress and must be documented or fixed; size/TTL/visibility are secondary.

---

### Cosmos DB
This package is a **read-side Change Feed consumer** (`BenzeneCosmosChangeFeedWorker<TDocument>`), not a client/repository. CLAUDE.md explicitly declines an egress package ("Cosmos DB is a database, not a transport… bring your own `CosmosClient`").

**[Out of scope by design] Write-side repository concerns** (Severity: N/A — deliberate non-goal)
- upsert/create, partition-key-on-write, RU/throughput, bulk, `TransactionalBatch`, ETag/optimistic concurrency, point-read vs query, item TTL — all belong to the write side Benzene intentionally doesn't provide. Not gaps. On the consumer side, partition keys are handled correctly and implicitly: leases are one-per-partition-key-range (correct SDK model).

**[MISSING] Latest-version change feed only — deletes not captured** (Severity: Low/Medium)
- `CosmosChangeFeedProcessorFactory` hardcodes `GetChangeFeedProcessorBuilderWithManualCheckpoint(...)` (latest-version mode: creates + updates only). Capturing deletes/intermediate versions requires all-versions-and-deletes mode (`GetChangeFeedProcessorBuilderWithAllVersionsAndDeletes`, container retention configured). A handler expecting delete reactions never sees them. Document the limitation; optionally offer an all-versions factory variant.

**[Known limitation] Host cancellation tokens unobserved; coarse batch-level checkpoint** (Severity: Low)
- SDK `StartAsync`/`StopAsync` take no token (host tokens unobserved); checkpointer ignores the per-item argument and checkpoints the whole batch (change feed has no per-document resume token). Documented honestly; acceptable.

**Verdict — Cosmos DB:** correctly scoped as a read-side change-feed consumer with faithful lease/checkpoint semantics; write-side "gaps" are deliberate non-goals; the one genuine in-scope limitation is latest-version-only (no delete capture).

---

Overall: no schema-validity bugs in Event Grid (required fields populated on both paths). Highest-impact issue is Queue Storage message encoding — egress default (plain) contradicts Benzene's own Functions ingress default (Base64), undocumented. Everything else is missing-feature breadth or deliberately-scoped-out write-side functionality.
