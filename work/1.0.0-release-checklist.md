# Benzene 1.0.0 Release Checklist

Goal: move off `-alpha` versioning so Benzene is safe for teams to
adopt in real projects, not just spikes/prototypes.

## 1. API surface review (do this first — everything else depends on it)

- [ ] List every public type/method across all packages (`Benzene.Http`,
      `Benzene.Aws.Lambda.Core`, `Benzene.Aws.Lambda.Kafka`, and any
      core package). A quick way: `dotnet build` with
      `<GenerateDocumentationFile>true</GenerateDocumentationFile>`
      and skim the generated XML doc for the public API list.
- [ ] For each public member, ask: "would I be comfortable if this
      exact signature had to stay stable for a year?" Once you hit
      1.0, changing any of this becomes a breaking change under
      semver — so this is the last free moment to rename, restructure,
      or remove things.
- [ ] Decide what's genuinely 1.0-ready vs. what should stay
      `-preview`/internal a bit longer (e.g. if Kafka support is
      newer/less battle-tested than Http, it's fine to ship
      `Benzene.Http` at 1.0 and keep `Benzene.Aws.Lambda.Kafka` at
      a later preview tag)

## 2. Versioning & compatibility policy

- [ ] Adopt semver (`MAJOR.MINOR.PATCH`) formally and document it
      (a short "Versioning" section in the README or a
      `VERSIONING.md`)
- [ ] Decide target framework support (e.g. .NET 8+ only, or wider)
      and state it explicitly
- [ ] Set up (or confirm) a `CHANGELOG.md` — start it now, even
      retroactively for past alpha versions, so 1.0.0 isn't the
      first entry

## 3. Test coverage gate

- [ ] Run coverage across `src/` and identify any public API paths
      with no test coverage — these are the highest-risk places for
      a 1.0 regression
- [ ] Make sure CI (GitHub Actions) runs the full suite on every PR,
      not just on merge to main, if it doesn't already
- [ ] Add a badge for this to the README once confirmed

## 4. Documentation gate

- [ ] Every public type intended for 1.0 has XML doc comments
      (shows up in IntelliSense — this matters a lot for adoption,
      since most people's first interaction with your API is
      autocomplete, not the docs site)
- [ ] `docs/` covers: getting started, core concepts (middleware
      pipeline, ports/adapters), and one guide per adapter package
- [ ] `examples/` projects build and run cleanly from a fresh clone
      (this is worth testing literally — clone into a scratch
      directory and try it)

## 5. Breaking change communication

- [ ] If 1.0 introduces any breaking changes vs. the last alpha,
      write a short migration note
- [ ] Tag the release clearly in GitHub Releases with release notes,
      not just a bare version bump

## 6. Publish

- [ ] Bump version, update `CHANGELOG.md`
- [ ] Publish to NuGet
- [ ] Create a GitHub Release with notes (this is also what gets
      picked up if the repo ever appears in "trending" or similar —
      worth writing well, not just "1.0.0 release")
- [ ] Update README badges/install instructions to remove any
      "prerelease" notes

## Notes on pace

This doesn't need to happen all at once. The API surface review
(step 1) is the only genuinely blocking step before you can even
plan the rest — everything downstream depends on knowing what's
actually going to be in the stable public API. Realistic path:
do step 1 as a dedicated pass (possibly a good task to run through
Claude Code in Plan Mode, given the scale — "review the public API
surface across all Benzene packages and flag anything that looks
unstable, inconsistent, or unfinished for a 1.0 release"), then
work through 2-6 incrementally.
